# 缺陷修复提案：启动脚本可靠重编译并重启 3201 服务

## 修复动机

当前 `start-3201.cmd` 会调用 `scripts/restart-3201.mjs`，但构建函数在检测到
`.next/BUILD_ID` 后直接返回，因此重复执行脚本不会重新编译代码。同时，停止旧服务后
没有显式验证监听端口已经释放，缺少对“旧服务确实退出”这一关键行为的端到端保证。

本次修复必须让脚本在已有服务监听 `0.0.0.0:3201` 时仍可可靠完成一次完整的
“停止旧服务、重新构建、启动新服务、确认可用”流程。

## 已确认的范围

- Windows 双击入口仍为根目录 `start-3201.cmd`。
- 只修改 `scripts/restart-3201.mjs` 及其测试，必要时对入口脚本做最小调整。
- 端口固定为 `3201`，监听地址固定为 `0.0.0.0`。
- 只允许关闭命令行可证明属于当前仓库的旧 Viewer 服务。
- 若端口由无关进程占用，报错退出，不强制结束该进程。

## 诊断假设（按优先级）

1. **已有构建产物使重新编译被跳过。** 若这是原因，移除 `BUILD_ID` 短路后，
   即使 `.next/BUILD_ID` 已存在也会执行一次真实的 `pnpm build`。
2. **旧进程结束与后续步骤之间缺少明确的释放确认。** 若这是原因，在 `taskkill`
   后等待旧 PID 消失且端口不再监听，可以消除重复启动时的竞争或阻塞。
3. **CMD 入口仍被子进程句柄保持。** 若这是原因，即使服务已监听，入口进程也不会
   在限定时间内退出；最终端到端验证会直接捕获该症状。

## 修复方案

脚本按以下固定顺序执行：

1. 检查 `3201` 的监听进程。
2. 若为当前仓库的旧服务，结束其进程树并等待端口释放；若为无关进程则安全退出。
3. 无条件执行 `pnpm build`，构建失败时不启动服务并返回非零退出码。
4. 以脱离 CMD 的后台进程启动 Next.js，参数为
   `start --hostname 0.0.0.0 --port 3201`。
5. 等待端口接受连接后让 CMD 成功退出，并保留现有 stdout/stderr 日志。

### 实施期补充决策：通过 CMD 调用 pnpm.cmd

首次真实验收发现，Node.js 22 在 Windows 上直接执行解析到的
`C:\Program Files\nodejs\pnpm.CMD` 会返回 `spawnSync ... EINVAL`，进程状态为
`null`；同一路径通过 `cmd.exe /d /c call` 调用时可正常输出 pnpm 11.6.0。

因此生产构建必须通过 Windows 命令解释器调用 `.cmd` 文件，并把参数作为独立参数
传递，兼容路径中的空格。构建失败信息同时优先保留底层 `spawn` 错误，避免再次只显示
`code unknown`。该调整不改变入口、端口或应用行为，只修正 Windows 进程边界。

选择“先停旧服务、再构建”会增加一次构建时长的停机窗口，但可避免运行中的
Next.js 同时读取正在被覆盖的 `.next` 目录。对于本地 Viewer 启动脚本，这是更安全、
更可预测的取舍。

## 非目标

- 不引入 PM2、Windows Service 或其他守护进程框架。
- 不改变应用 API、数据模型、前端逻辑或依赖版本。
- 不清理 `.next`、`node_modules`、`.runtime` 或用户现有的 `.claude/`。

## 验收标准

- 即使 `.next/BUILD_ID` 已存在，每次执行入口脚本仍会重新构建。
- 已监听 `0.0.0.0:3201` 的当前仓库旧服务被正确关闭，旧 PID 不再存在。
- 新服务使用不同 PID 监听 `0.0.0.0:3201`，并可通过 HTTP 访问。
- `start-3201.cmd` 在服务就绪后自行退出，不被后台服务阻塞。
- 无关进程占用 `3201` 时不会被结束。
- pnpm 位于含空格路径时，构建命令仍能执行；若无法启动，错误信息包含底层原因。

## 影响范围

- API 契约：无变化。
- 数据模型与数据库：无变化。
- 配置与部署：仅调整 Windows 本地启动/重启流程。
- 向后兼容：保留入口文件名、端口、监听地址和日志路径。
