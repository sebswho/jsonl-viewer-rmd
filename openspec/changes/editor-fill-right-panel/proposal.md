# 缺陷修复提案：编辑框填满右侧纵向区域

## 修复动机

用户通过记录右键菜单进入编辑模式后，右侧内容区高度为 736.5px，但 textarea 仅有
固定的 320px 高度，底部留下约 400.5px 空白。编辑大量或格式化后的 JSON 时，实际可用
空间没有被利用。

已使用无头 Chrome 在 1440×900 视口复现真实交互路径：上传 JSONL、右键记录、进入
编辑模式，然后比较 textarea 与右侧内容容器的边界。当前自动断言要求底部空白不超过
24px，实际为 400.5px，稳定失败。

## 已确认的范围

- 仅调整右侧“编辑记录”状态，不改变格式化、原始和树形查看状态。
- 编辑框自动填满错误提示之外的剩余纵向空间。
- JSON 无效时，错误提示继续显示在编辑框下方。
- 取消 textarea 的手动纵向缩放，避免用户破坏自动填充布局。
- 不改变编辑、校验、保存或取消的数据行为。

## 诊断结论

1. `.editor` 只有 `min-height: 320px`，计算后的 flex 值为 `0 1 auto`，不会增长。
2. 编辑态复用了普通块布局的 `.detailContent`，即使给 textarea 增加 `flex: 1`，父容器
   也不会按剩余高度分配空间。
3. 直接把所有 `.detailContent` 改成 flex 会影响其他查看模式，因此需要编辑态专用类。

## 修复方案

1. 编辑态的内容容器同时使用现有 `.detailContent` 和新增 `.editorContent`。
2. `.editorContent` 使用纵向 flex 布局并设置 `min-height: 0`，保持错误提示参与布局。
3. `.editor` 使用 `flex: 1`、`min-height: 0` 和 `resize: none`，占满剩余空间。
4. 保留现有内边距、颜色、校验边框和滚动能力。

选择专用修饰类而不是全局修改 `.detailContent`，可以把 CSS 影响限定在编辑状态，避免
单场景需求污染格式化、原始和树形视图。

## 验收标准

- 在 1440×900 视口进入有效 JSON 编辑态后，textarea 底边与右侧内容容器底边的差值
  不超过 24px。
- 编辑框保持内部滚动，并且不能手动纵向缩放。
- JSON 无效时错误提示仍位于编辑框下方且可见。
- 现有编辑、保存、取消、删除和查看模式测试继续通过。

## 非目标与影响范围

- API、数据模型、文件序列化和持久化行为：无变化。
- 依赖、环境变量和部署拓扑：无变化。
- 响应式布局仅验证现有桌面使用场景，不重新设计左右面板结构。
