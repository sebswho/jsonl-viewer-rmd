# ADR 0001: 使用 Chromium 文件句柄实现会话恢复

## 状态

已接受

## 背景

用户需要在刷新或重新打开页面后恢复已打开文件和工作上下文。目标文件通常大于 100 MB，不适合默认持久化内容快照到浏览器本地存储。

## 决策

本次变更采用 Chromium `showOpenFilePicker()` + `FileSystemFileHandle` 作为可恢复文件的唯一持久化机制：

- 通过原生文件选择器打开的文件被视为可恢复文件。
- 会话持久化保存文件句柄和界面状态，不保存大文件内容快照。
- 恢复时重新从句柄读取文件内容并重新解析。
- 拖拽或普通文件输入打开的文件作为临时文件处理，不承诺跨会话恢复。

## 理由

1. 适配大文件场景，避免 IndexedDB/OPFS 大体积缓存带来的配额和性能风险。
2. 方案与用户运行环境一致，用户已明确仅需要 Chromium 浏览器支持。
3. 保留现有拖拽/文件输入体验，同时新增一条可恢复的标准打开路径。

## 替代方案

### 方案 A：把文件内容快照写入 IndexedDB

拒绝原因：

- 对 100 MB+ 文件不稳定，配额不可预测。
- 恢复时需要重新载入大块文本，首屏和内存压力明显增加。
- 会把本次需求扩展成浏览器端大文件存储系统。

### 方案 B：仅保存文件名和界面状态

拒绝原因：

- 不能真正恢复文件内容，用户价值不足。

## 后果

- 会话恢复能力依赖 Chromium 能力和用户曾授予的文件访问权限。
- 非 Chromium 浏览器或未获得可持久化句柄的文件，只能维持当前会话可见。
